# Investigation

## Objective

The objective of this phase is to investigate the SSH brute-force alert generated by Wazuh, validate whether the activity was malicious, determine whether any unauthorized access was obtained, and document the findings.

# Step 1 – Review the Wazuh Alert

## Objective

Review the alert generated by Wazuh to identify the key indicators associated with the detected activity.

The alert provides important information including:

- Rule ID
- Alert Level
- Description
- Timestamp
- Source IP Address
- Target Username
- Agent Name

### Example

| Field | Example |
|--------|----------|
| Rule ID | 5710 |
| Alert Level | 5 |
| Description | Multiple SSH authentication failures |
| Agent | Ubuntu Endpoint |
| Source IP | 10.0.2.15 |
| Username | baduser |

### Observation

The alert indicates multiple failed SSH authentication attempts originating from a single IP address within a short period.

This pattern is consistent with an SSH brute-force attack.

---

# Step 2 – Review Authentication Logs

## Objective

Verify the authentication attempts recorded by the operating system.

Run:

```bash
sudo grep "Failed password" /var/log/auth.log
```

or

```bash
sudo cat /var/log/auth.log
```

### Example Output

```text
Jul 1 21:47:11 wazuhagent123-VirtualBox sshd[9805]: Failed password for invalid user baduser from 10.0.2.15 port 56348 ssh2
```

### Observation

The authentication logs confirm multiple failed SSH login attempts.

All failed attempts originated from the same source IP address.

The logs identify the targeted username as `baduser`, which is an invalid user account.

---

# Step 3 – Check for Successful Logins

## Objective

Determine whether any authentication attempt was successful.

Run:

```bash
sudo grep "Accepted password" /var/log/auth.log
```

### Observation

No successful SSH login events were found.

There is no evidence that the attacker successfully authenticated to the system.

---

# Step 4 – Check Active Sessions

## Objective

Verify whether any unauthorized users currently have an active session.

Run:

```bash
who
```

### Observation

Only the legitimate user session was active.

No unexpected or unauthorized SSH sessions were identified.

---

# Step 5 – Review SSH Service Logs

## Objective

Review SSH daemon events to validate the authentication activity.

Run:

```bash
sudo journalctl -u ssh
```

Look for events such as:

- Failed password
- Accepted password
- Invalid user
- Connection closed
- Disconnected from authenticating user

### Example

```text
Failed password for invalid user baduser

Connection closed

Disconnected from authenticating user
```

### Observation

The SSH service logs correlate with the events recorded in `/var/log/auth.log`.

No `Accepted password` events were identified.

This confirms that all authentication attempts were unsuccessful.

# Analyst Assessment

The investigation identified multiple failed SSH authentication attempts originating from a single source IP address targeting the invalid user account `baduser`.

Authentication logs and SSH service logs confirmed repeated failed login attempts with no successful authentication events.

A review of active sessions found no unauthorized users connected to the system. Additionally, there was no evidence of privilege escalation, persistence mechanisms, or post-compromise activity.

Based on the available evidence, the activity is classified as an unsuccessful SSH brute-force attack that was successfully detected by Wazuh.

# Conclusion

The investigation confirmed that Wazuh successfully detected multiple failed SSH authentication attempts originating from the attacker VM.

Analysis of the authentication logs, SSH service logs, and active user sessions verified that no successful authentication occurred. No evidence of privilege escalation, persistence, or unauthorized access was identified.

The endpoint remained uncompromised, and no additional malicious activity was observed following the attack. The alert was accurately classified as an unsuccessful SSH brute-force attack.
