# Phase 4 - Investigation

## Objective

The objective of this phase is to investigate the SSH brute-force alert generated by Wazuh, validate whether the activity is malicious, determine if any unauthorized access was obtained, and document the findings.

---

# Investigation Workflow

```text
Wazuh Alert
      │
      ▼

      ▼
Review Authentication Logs
      │
      ▼

      │
      ▼

      │
      ▼

      │
      ▼

      ▼

```

# Step 1 - Review the Wazuh Alert

## Objective

Begin by reviewing the alert generated by Wazuh.

The alert contains valuable information such as:

- Rule ID
- Alert Level
- Description
- Timestamp
- Source IP
- Username
- Agent Name

### Example

| Field | Example |
|--------|----------|
| Rule ID | 5710 |
| Level | 5 |
| Description | Multiple SSH authentication failures |
| Agent | Ubuntu Endpoint |
| Source IP | 10.0.2.15 |
| Username | baduser |

---

### Observation

The alert indicates that multiple failed SSH login attempts were detected within a short period from a single source IP address.

This behavior is consistent with an SSH brute-force attack.

---

# Step 2 - Review Authentication Logs

Authentication logs contain the actual login attempts.

Run

```bash
sudo cat /var/log/auth.log
```

or

```bash
sudo grep "Failed password" /var/log/auth.log
```

Example Output

```text
wazuhagent123@wazuhagent123-VirtualBox:/$ sudo grep -a "Failed password" /var/log/auth.log
Jul  1 21:47:11 wazuhagent123-VirtualBox sshd[9805]: Failed password for invalid user baduser from 10.0.2.15 port 56348 ssh2


## Observation

The logs confirm repeated failed authentication attempts.

All login attempts originated from the same IP address.

No successful authentication entries were observed during these events.

---


### Observation

No successful login originating from the attacker's IP address was identified.

This indicates the brute-force attack was unsuccessful.

---

# Step 4 - Check Active Sessions

Verify whether any suspicious users are currently connected.

Run

```bash
who
```


### Observation

Only the legitimate user session was active.

No unexpected SSH sessions were found.

---

# Step 5 - Review SSH Service Logs

Review SSH service events.

Run

```bash
journalctl -u ssh
```

Look for

- Failed password
- Accepted password
- Invalid user
- Connection closed

Example

```text
Failed password for baduser

Connection closed

Disconnected from authenticating user
```

### Observation

The SSH service logs match the events recorded in /var/log/auth.log.

No "Accepted password" events were found.

This confirms that authentication never succeeded.

# Analyst Assessment

The repeated authentication failures originated from a single host attempting to authenticate using an existing user account.

Authentication logs confirmed multiple failed password attempts with no successful login events.

No evidence of privilege escalation, persistence mechanisms, or unauthorized access was identified.

Based on the available evidence, the activity is classified as an unsuccessful SSH brute-force attack.

# Conclusion

The investigation verified that Wazuh successfully detected multiple failed SSH authentication attempts originating from the attacker VM.

Analysis of authentication logs, SSH service logs, and active user sessions confirmed that the attacker did not successfully authenticate to the endpoint.

The endpoint remained uncompromised, and no additional malicious activity was observed following the attack.
